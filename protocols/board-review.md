# 董事会审议协议

本协议用于把输入材料转化为结构化《董事会建议书》。执行时优先保证判断质量和可执行性，不追求人物表演感。协议可被审议模式裁剪，但所有模式都必须保留结构化拆解、证据强度、反证实验和 Go / Pivot / No-Go。

## 审议模式裁剪规则

- `quick_triage`：保留第 0、1、3、5、6 轮；委员会合议可以压缩为三组摘要。
- `deep_board_review`：执行所有轮次。
- `red_team`：强化第 2、5、6 轮；机会分析从简，失败路径和停止条件必须详写。
- `pre_mortem`：以“项目已经失败”为前提执行第 0、2、3、5、6 轮。
- `investment_committee`：强化商业质量、资本效率、护城河、下行风险和周期判断。
- `product_discovery`：强化用户任务、价值风险、可用性风险、发现实验和可追溯证据。
- `go_to_market_review`：强化目标客户、渠道、定价、竞争定位、销售阻力和进入节奏。

最低质量门槛：

- 必须给出材料拆解，不能直接进入建议。
- 必须区分事实、推断和假设。
- 必须列出能推翻当前方案的反证实验。
- 必须给出 Go / Pivot / No-Go 倾向和触发条件。
- 必须说明证据强度，不得把低置信假设写成事实。

## 第 0 轮：输入材料结构化拆解

先生成 brief，不做评价：

- 输入类型：`product_requirement`、`project_plan` 或 `business_plan`。
- 摘要：忠实复述材料，不补充外部事实。
- 目标：材料显式目标；不明确时标注为假设。
- 利益相关方：用户、购买者、执行团队、审批者、受影响团队。
- 当前替代方案：用户或组织现在如何解决问题。
- 商业/产品机制：价值如何产生、交付、收费或复用。
- 执行计划：里程碑、资源、依赖、约束。
- 关键假设：业务、产品、市场、执行、组织、财务假设。
- 约束：时间、预算、人员、技术、合规、渠道。
- 未知信息：影响判断但材料没有提供的信息。

## 第 1 轮：Persona 独立审阅

每个 persona 只基于 brief 和原始输入独立输出：

- 判断：一句话结论。
- 机会：最值得抓住的 1-3 个点。
- 风险：最可能导致失败的 1-3 个点。
- 证据需求：做判断前必须补的证据。
- 反证实验：能推翻当前方案的验证动作。
- 质询动作：对同组或跨组观点提出的挑战。
- 建议：下一步行动或决策条件。

规则：

- 不引用其他 persona。
- 不编造人物原话或未验证事实。
- 不写闲聊式对话。
- 若材料不足，说明不确定性。

## 第 2 轮：委员会合议

每个委员会合并本组 persona 意见，输出：

- 强共识。
- 弱共识。
- 组内分歧。
- 少数派警告。
- 最重要的反证实验。
- 最重要的建议。
- 对最终 Go / No-Go / Pivot 的倾向。

合议时保留差异，不能把所有观点压成平均话术。

## 第 3 轮：跨委员会综合

比较所有委员会结论：

- 哪些判断形成强共识。
- 哪些判断冲突，冲突来自事实假设、价值排序还是时间尺度。
- 哪些少数派警告值得保留。
- 哪些行动建议可以合并。
- 哪些证据足够强，哪些只是低置信假设。
- 哪些失败路径有毁灭性下行，必须先处理。

## 第 4 轮：董事会建议书

使用 `templates/board-memo.md` 输出最终报告。报告必须先给结论，再给证据和分歧，最后给行动建议。

必须插入 3 个 Mermaid 图表：

- 输入材料结构化拆解后：用 `flowchart LR` 输出价值链 / 工作流图，说明从客户问题到交付物再到商业结果的路径。
- 各委员会结论后：用 `flowchart TD` 输出董事会审议信号图，说明各委员会倾向、强共识、关键分歧和少数派警告。
- 30 / 60 / 90 天行动方案后：用 `flowchart LR` 输出路线图，说明 30 天、60 天、90 天分别要验证什么，以及触发 Go / Pivot / No-Go 的门槛。

图表规则：

- Mermaid 只表达报告中已经给出的事实、假设、判断和行动，不新增未验证数据。
- 节点文字使用短句，必要时用“待验证”标出证据缺口。
- 不用复杂 Mermaid 语法，优先保证 Markdown 渲染稳定。

结论等级：

- `Go`：方向值得推进，但仍需按建议控制风险。
- `Pivot`：目标可能成立，但当前方案、定位、路径或节奏需要调整。
- `No-Go`：关键假设不成立、下行风险过大，或材料不足以支持投入。

## 第 5 轮：附录 A 证据包

把最终报告中最重要的判断拆成证据包。每条证据包必须包含：

- `claim`：判断内容。
- `claim_type`：fact、inference 或 assumption。
- `evidence_source`：来自用户材料、审议推断、缺失证据或待补材料。
- `confidence`：high、medium 或 low。
- `counterevidence`：能削弱该判断的反向证据。
- `disproof_test`：下一步最小反证实验。

## 第 6 轮：附录 B 待验证假设

记录后续复盘必须追踪的关键假设。每条假设必须包含：

- 假设内容。
- 所属类型：产品、市场、执行、组织、财务、合规或技术。
- 当前证据。
- 反证方式。
- 30 / 60 / 90 天检查点。

## 第 7 轮：附录 D 决策记录

为每次深度审议生成可归档的决策记录。记录必须能独立说明当时的建议、关键假设、待验证证据和下次复盘时间，不依赖聊天上下文。
