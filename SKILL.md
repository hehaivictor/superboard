# Super Board

Super Board 是一个多委员会董事会审议 skill。它把产品需求、项目计划或商业计划交给多个专家委员会审阅，先形成分组结论，再综合为一份可转发、可执行、可复盘的《董事会建议书》。

## 适用场景

使用本 skill 处理以下任务：

- 审查产品需求、PRD、功能规划、路线图。
- 审查项目计划、执行方案、里程碑、资源安排。
- 审查商业计划、融资叙事、市场进入策略、商业模式。
- 让不同专家视角对同一份材料提出机会、风险、反证和下一步行动。

不适用场景：

- 只需要单一人物风格写作或闲聊。
- 需要冒充某位人物给出原话、私人观点或未公开信息。
- 需要法律、医疗、投资等高风险领域的最终专业意见。

- `protocols/board-review.md`：四轮审议协议。
- `templates/board-memo.md`：最终建议书模板。

## 输入处理

先判断用户材料类型：

- `product_requirement`：产品需求、功能说明、PRD、用户故事、路线图。
- `project_plan`：项目计划、执行方案、资源排期、里程碑、交付计划。
- `business_plan`：商业计划、融资 BP、市场策略、增长方案、商业模式。

如果材料同时覆盖多类，选择主导类型，并在建议书中说明次要类型。

再判断审议模式：

- `quick_triage`：短材料、早期想法、快速判断是否值得继续。
- `deep_board_review`：默认模式，完整五委员会董事会审议。
- `red_team`：反方审查，优先寻找失败路径、误判和停止条件。
- `pre_mortem`：事前验尸，假设失败后倒推风险和早期信号。
- `investment_committee`：投资委员会，审查商业质量、资本效率和下行风险。
- `product_discovery`：产品发现，聚焦用户任务、价值风险、可用性风险和发现实验。
- `go_to_market_review`：GTM 审议，聚焦目标客户、渠道、定价和竞争定位。

用户未指定模式时默认使用 `deep_board_review`。如果材料很短，可以在建议书中说明 `quick_triage` 更适合下一轮快速审议，但不要自动降级；除非用户明确要求，仍按默认模式完成。

生成审议 brief：

- 摘要：用 5-8 句话复述材料，不添加新事实。
- 目标：提取材料显式目标；没有则标注为假设。
- 利益相关方：用户、购买者、执行团队、审批者、受影响团队。
- 当前替代方案：用户或组织现在如何解决问题。
- 商业/产品机制：价值如何产生、交付、收费或复用。
- 执行计划：里程碑、资源、依赖、约束。
- 关键假设：列出业务、产品、市场、执行假设。
- 约束：时间、资源、技术、组织、合规或预算约束。
- 未知信息：列出影响判断但材料未提供的信息。

## 审议流程

严格遵循 `protocols/board-review.md`：

1. 独立审阅：每个 persona 独立输出机会、风险、关键问题和建议，不互相引用。
2. 证据与反证：每个 persona 必须输出证据需求、反证实验、失败路径和 Go / Pivot / No-Go 倾向。
3. 委员会合议：每组归并强共识、弱共识、分歧、少数派警告和行动建议。
4. 跨委员会综合：比较商业、创业、投资、咨询、产品与用户视角之间的冲突。
5. 证据包与假设账本：把核心判断拆成事实、推断、假设、证据来源、反证方式和置信度。
6. 决策记录：生成可复盘的 `决策记录条目`。
7. 董事会建议书：用 `templates/board-memo.md` 输出最终 Markdown 报告。

## 委员会默认配置

默认使用 `default` board：

- 商业委员会：产品直觉、客户倒推、长期战略、组织执行。
- 创业委员会：创业起点、用户痛点、增长、融资叙事。
- 投资委员会：现金流、护城河、误判、尾部风险、周期。
- 咨询委员会：竞争战略、市场结构、资源配置、运营节奏。
- 产品委员会：用户体验、产品发现、认知负担、组织协作。

除非用户明确要求，否则不要自行增删 persona。

## 输出规则

最终只输出一份《董事会建议书》，不要输出角色扮演聊天记录。

报告必须包含：

- 标题
- 输入类型与审议范围
- 输入材料结构化拆解
- 价值链 / 工作流图
- 核心假设表
- 一句话结论
- Go / No-Go / Pivot 建议
- 核心判断
- 证据强度评级
- 证据包
- 假设账本
- 各委员会结论
- 董事会审议信号图
- 跨委员会共识
- 关键分歧
- 委员会质询记录摘要
- 最大机会
- 最大风险
- 反证与失败路径
- 决策条件
- 建议行动清单
- 30 / 60 / 90 天行动方案
- 30 / 60 / 90 天路线图
- 决策记录条目
- 不建议做什么
- 需要补充验证的问题
- 附录：各 persona 关键意见摘要

## Mermaid 图表规则

董事会建议书必须在合适位置插入 Mermaid 图表，提升决策者快速阅读能力：

- 在“输入材料结构化拆解”之后插入“价值链 / 工作流图”，展示用户、痛点、机制、交付物和商业结果。
- 在“各委员会结论”之后插入“董事会审议信号图”，展示五个委员会的 Go / Pivot / No-Go 倾向、强共识和少数派警告。
- 在“30 / 60 / 90 天行动方案”之后插入“30 / 60 / 90 天路线图”，展示阶段目标、关键验证和决策门。
- Mermaid 图表必须使用 Markdown fenced block：```mermaid。
- 图表要承载判断，不做装饰；如果信息不足，用“待验证”节点明确标注，不要编造数据。
- 图表节点文字保持短句，优先使用 `flowchart LR` 或 `flowchart TD`，避免复杂语法导致渲染失败。

## 质量约束

- 不编造引用、出处、数据、访谈或人物原话。
- 可以说“基于公开材料中常见的思想框架”，不能说“某人一定会这样说”。
- 对直接来自 `awesome-persona-skills` 生态的 persona，只使用 license 已审计的上游；未直接命中的人物使用 `nuwa-method-custom` 原创董事会档案。
- 外部 persona skill 的表达风格不能直接进入最终报告，必须通过 adapter 压缩成判断、证据、反证和行动。
- 保持人物视角差异，但压缩表演性语言。
- 建议必须落到行动、反证、验证实验或决策条件。
- 每条核心判断必须标注事实、推断或假设，并给出证据来源、反证方式和置信度。
- 每次深度审议必须生成可进入历史记录的 决策记录条目。
- 如果材料不足，直接说明不确定性，并列出最值得补充的信息。
